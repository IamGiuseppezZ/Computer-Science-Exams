ESAME DI SISTEMI OPERATIVI - PROVA DI LABORATORIO
TITOLO: Analizzatore di Spazio Disco (Pipeline con Condizioni e Semafori)

DESCRIZIONE DEL PROBLEMA
Si realizzi un'applicazione multithread in linguaggio C (usando la libreria 
<pthread.h>) in grado di calcolare in modo concorrente lo spazio occupato 
dai file contenuti all'interno di una o più directory.

Il programma deve ricevere da riga di comando l'elenco delle directory 
da scansionare:
Uso: ./programma <dir1> <dir2> ... <dirN>

ARCHITETTURA DEI THREAD E PROTOCOLLO DI COMUNICAZIONE
Il sistema deve implementare una pipeline a 3 stadi che fa uso sia di 
Semafori (per la gestione di buffer) sia di Variabili di Condizione (per la 
sincronizzazione a passo singolo).

1. THREAD "DIR" (Scanner Directory - Produttori Stadio 1):
   - Viene creato un thread DIR per ogni directory passata da terminale.
   - Il thread legge il contenuto della propria directory.
   - Per ogni file regolare (DT_REG) trovato, costruisce il percorso completo 
     (es. "dir_file/immagine.jpg") e lo inserisce all'interno di un Buffer 
     Circolare Condiviso (di dimensione BUFFER_SIZE = 3).
   - L'accesso al buffer circolare deve essere governato esclusivamente 
     tramite Semafori (modello Produttore/Consumatore standard).

2. THREAD "STAT" (Analizzatore - Consumatore Stadio 1 / Prod. Stadio 2):
   - Esiste un SOLO thread STAT.
   - Preleva ciclicamente i percorsi dei file dal Buffer Circolare.
   - Utilizza la system call stat() per calcolare la dimensione in byte 
     del file estratto.
   - Comunica il risultato (Nome e Dimensione) al thread MAIN. 
   - VINCOLO SUL PASSAGGIO DATI: Questo passaggio dati NON deve usare 
     semafori o array. Deve avvenire usando una singola variabile strutturata 
     (es. struct Info_File file_main) condivisa tra STAT e MAIN. La 
     sincronizzazione (evitare che STAT sovrascriva i dati prima che MAIN li 
     abbia letti) deve avvenire rigorosamente tramite l'utilizzo di Mutex 
     e Variabili di Condizione (pthread_cond_t) secondo un modello a 
     sincronizzazione stretta (Ping-Pong).

3. THREAD "MAIN" (Aggregatore - Consumatore Finale):
   - Esiste un SOLO thread MAIN (che non è il main() di avvio).
   - Attende sulla variabile di condizione che STAT gli renda disponibile 
     i dati di un nuovo file.
   - Stampa a video l'operazione in corso e accumula la dimensione in una 
     variabile contenente il totale (parziale) globale.
   - Segnala a STAT che la struttura condivisa è tornata libera e si rimette 
     in attesa.

TERMINAZIONE E PULIZIA
- L'utilizzo dell'attesa attiva (busy waiting) comporterà l'annullamento 
  della prova.
- Nessun thread deve rimanere in situazione di Deadlock. Si consiglia 
  l'uso di una stringa sentinella (es. "EOF_TERMINATE") iniettata nel buffer 
  dall'ultimo thread DIR che conclude il suo lavoro, per notificare a STAT 
  di terminare a sua volta (e STAT farà lo stesso con MAIN).
- Tutte le risorse dinamiche e le primitive di sincronizzazione devono essere 
  liberate dal processo principale prima dell'uscita. Alla fine, il MAIN 
  stamperà la dimensione in byte totale calcolata.
